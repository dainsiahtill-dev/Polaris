## 🧠 核心数据流（收敛后）

### 1. 工具调用数据流

```
LLM Response
    │
    ▼ parse_tool_calls() [统一入口]
list[ToolCall] (canonical)
    │
    ▼ ToolExecutorPort.execute()
ToolExecutionResult
    │
    ▼ MessageBus.emit("tool.call")
TypedEventBusAdapter
    │
    ▼ OmniscientAuditBus (审计)
    │
    ▼ AuditEvent (Pydantic)
```

### 2. 跨Cell通信数据流

```
Cell A (roles.kernel)
    │
    ▼ ContextBudgetPort (kernelone Port)
Cell B (llm.control_plane)
    │
    ▼ 通过 Port 接口调用
    │
    ▼ 无 internal 直接导入
```

### 3. 事件总线数据流

```
Producer
    │
    ▼ MessageBus.publish(topic, payload)
TypedEventBusAdapter
    │
    ├─→ EventRegistry (typed events)
    │
    ├─→ OmniscientAuditBus (audit events)
    │
    └─→ NatsBusAdapter (external)
```

## 🏗️ 技术选型理由

### 1. 为什么废弃 STANDARD_TOOLS？

| 标准 | STANDARD_TOOLS | _TOOL_SPECS |
|------|----------------|-------------|
| 工具数 | 16 | 40 |
| repo_* 工具 | 无 | 有 |
| 状态 | deprecated | active |
| 引用点 | executor 降级路径 | 正常路径 |

**结论**: _TOOL_SPECS 是实际运行的真相源，STANDARD_TOOLS 是历史遗留

---

### 2. 为什么 MessageBus 作为规范总线？

| 总线 | 特性 | 评估 |
|------|------|------|
| MessageBus | Actor 模型，dead letter queue | **稳定，生产验证** |
| EventRegistry | Typed events，wildcards | 类型安全，但新 |
| OmniscientAuditBus | Priority queue，熔断 | 审计专用，不应通用 |

**结论**: MessageBus 已验证稳定，其他通过 adapter 桥接

---

### 3. 为什么统一 Topic 为 "tool.call"？

| 格式 | 位置 | CloudEvents 兼容 |
|------|------|------------------|
| tool_call | uep_contracts | ❌ underscore |
| tool.call | nats_config | ✅ dot notation |
| tool-execution | audit_events | ❌ hyphen |

**结论**: dot notation 符合 CloudEvents/NATS 最佳实践

---

### 4. 为什么 Port 接口定义在 kernelone？

| 层级 | 职责 | Port 定义位置 |
|------|------|--------------|
| kernelone | 核心基础设施 | **规范接口定义** |
| cells | 业务逻辑 | 实现 Adapter |
| infrastructure | 底层适配 | 具体实现 |

**结论**: kernelone 是核心层，Port 接口应在此定义，cells 实现适配
