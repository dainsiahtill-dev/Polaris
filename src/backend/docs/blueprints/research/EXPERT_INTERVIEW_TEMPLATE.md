# 专家面试问题模板

**研究会**: ContextOS & TurnEngine 深度研究计划  
**面试目标**: 评估候选人的专业能力、问题解决能力和协作精神

---

## 1. 通用问题 (所有候选人)

### 1.1 背景与动机

1. **自我介绍** (2分钟)
   - 请简要介绍您的专业背景
   - 为什么对 ContextOS/TurnEngine 感兴趣？

2. **项目经验**
   - 描述一个您参与过的类似系统（AI Agent框架/对话系统/事件溯源系统）
   - 您在其中的具体贡献是什么？
   - 遇到了哪些挑战？如何解决的？

3. **研究动机**
   - 您如何看待当前AI Agent系统的可靠性问题？
   - 您认为ContextOS的核心理念是什么？

### 1.2 技术能力

4. **代码审查**
   ```python
   # 给候选人看这段代码
   def compress(self, target_tokens: int, llm=None) -> ContextOSProjection:
       # ...
       for event in self.active_window:
           if event.route.upper() == "PATCH":
               score += 10.0
           if event.route.upper() == "ARCHIVE":
               score -= 5.0
   ```
   - 这段代码有什么问题？
   - 如何改进？

5. **系统设计**
   - 设计一个断路器系统，检测LLM的重复工具调用
   - 需要哪些指标？阈值如何设定？
   - 如何防止误报？

### 1.3 协作与沟通

6. **冲突处理**
   - 如果与其他专家在技术方案上有分歧，您会如何处理？
   - 举例说明您过去的冲突解决经验

7. **时间管理**
   - 8周的全职投入，您的时间安排如何？
   - 是否有其他承诺可能影响参与？

## 2. 专业组问题

### 2.1 ContextOS 专家组

#### 事件溯源架构师 (C-01)

**Q1**: 审计以下事件序列，发现问题：
```json
[
  {"seq": 100, "event": "llm_call_start", "iteration": 5},
  {"seq": 101, "event": "tool_call", "iteration": null},
  {"seq": 102, "event": "tool_result", "iteration": null},
  {"seq": 103, "event": "llm_call_end", "iteration": 5}
]
```
- 问题是什么？
- 如何修复？
- 修复后如何验证？

**Q2**: 序列号达到 922606 后大幅下降至 66536，这可能是什么原因？
- 这是bug吗？还是正常行为？
- 如何确保轮转时不丢失事件？
- 设计一个更好的序列号方案

**Q3**: 设计一个"因果链追踪"系统，能够回答：
- "这个tool_result是由哪个llm_call触发的？"
- "本次turn的所有事件有哪些？"
- "跨turn的事件依赖关系是什么？"

#### 上下文压缩算法专家 (C-02)

**Q1**: 评估当前的Turn-Block压缩策略：
```python
# 当前实现
if event.source_turns and any(t == current_turn for t in event.source_turns):
    turn_key = current_turn
```
- 这个策略的优缺点？
- 什么情况下会失效？
- 设计一个改进方案

**Q2**: 如何量化"压缩质量"？
- 设计3-5个可测量的质量指标
- 如何自动评估这些指标？
- 压缩质量的"及格线"是多少？

**Q3**: 紧急压缩触发的思考：
- 当前阈值是50事件，触发后token目标减半
- 这个策略合理吗？
- 设计一个自适应阈值算法

#### 意图切换研究员 (C-03)

**Q1**: 评估当前的意图检测启发式：
```python
view_verbs = {"看", "分析", "检查", "read", "analyze"}
write_verbs = {"写", "创建", "修改", "write", "create"}
```
- 这个方法的局限性？
- 给出5个会失败的案例
- 设计一个基于embedding的改进方案

**Q2**: 意图切换的边界案例：
- "看下这个bug，然后修复它" → 单次指令包含两个意图，如何处理？
- "不对，我不是要创建文件，是要修改" → 用户纠正，如何回滚？
- "先分析A，再分析B，然后创建C" → 多步骤意图，如何追踪？

**Q3**: 设计一个"意图切换满意度"评估框架：
- 如何知道我们的切换是否正确？
- 需要收集什么数据？
- 如何在没有用户反馈的情况下评估？
