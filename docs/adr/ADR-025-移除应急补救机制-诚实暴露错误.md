# ADR-025: 移除应急补救机制，诚实暴露错误

## 状态
- **日期**: 2026-03-11
- **状态**: 已采纳
- **作者**: Claude Code

## 背景

### 问题现象
在压测过程中，Director 阶段频繁出现"卡住"现象：
- 首轮 Director 返回格式失败（未找到有效的 JSON 或补丁）
- 适配器自动触发"应急写入"，产生 67 行模板代码
- 压测门槛要求 80 行，触发 `sparse_output_detected`
- 进入 120s 的 LLM 重试等待，期间只有 heartbeat 日志
- 用户中断后，run 停留在 `quality_gate` 状态

### 根本原因
**应急补救机制掩盖了真实问题**：
1. LLM 输出格式错误时，系统自动写入模板代码
2. 模板代码与项目需求无关（通用 Ledger/Storage 模板）
3. 产生"有代码但质量不合格"的虚假进度
4. 重试等待时间过长，用户体验极差

## 决策

### 决策 1: 彻底移除应急补救机制
**原则**: 写代码就是写代码，不需要应急补救。

**删除内容**:
- `_execute_emergency_write_plan()` 的所有调用点（4 处）
- 首轮失败后的自动应急写入
- 重试失败后的自动应急写入
- 稀疏检测后的自动应急写入
- 稀疏重试后的自动应急写入

### 决策 2: 格式失败直接抛出异常
**原则**: 绝对诚实，不掩盖错误。

**新增异常**:
```python
# 首轮格式验证失败
raise RuntimeError(f"Director LLM 格式验证失败: {raw_error}")

# 重试后仍失败
raise RuntimeError(f"Director LLM 重试后仍格式验证失败: {retry_raw_error}")

# 稀疏输出
raise RuntimeError(f"Director 输出过于稀疏: files={n}/{min_files}, lines={n}/{min_lines}")

# 稀疏重试后仍无产出
raise RuntimeError(f"Director 稀疏重试后仍无有效代码产出")
```

### 决策 3: 重试策略上移
**原则**: 由上层（Factory/PM）决定重试策略。

**Factory 层重试**:
```python
max_retries = context.get("director_max_retries", 2)
while retry_count <= max_retries:
    result = await execute_director_run(...)
    if should_retry(result):
        await asyncio.sleep(exponential_backoff)
        retry_count += 1
    else:
        break
```

## 后果

### 正面
1. **问题暴露**: LLM 格式问题无法被掩盖，迫使质量改进
2. **快速失败**: 不再等待 120s 重试，立即返回错误
3. **责任清晰**: Director 只管执行，Factory 管重试策略
4. **用户可控**: 可通过 `director_max_retries` 参数控制重试

### 负面
1. **压测失败率上升**: 短期内 Director 阶段失败会增加
2. **需要 LLM 调优**: 必须改进 prompt 或模型选择
3. **日志量增加**: 错误信息会更频繁出现

## 教训

### 设计原则
1. **诚实优于容错**: 不要假装成功，要暴露真实问题
2. **不要自动补救**: 自动写入的代码几乎总是错的
3. **分层责任**: 底层抛出异常，上层决策重试
4. **快速失败**: 长等待比立即失败更糟糕
